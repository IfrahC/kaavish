\section{Project Plan}
\subsection{Objectives}
The goal of this project is to design and implement an Intelligent Scheduling and Performance Analytics Platform for Code School. The big picture is a centralized web-based CMS that automates administrative workflows, enhances scheduling efficiency, and provides AI-driven instructor evaluation reports based on classroom information.

Outside of the traditional CMS components, key objectives include:
\begin{itemize}
    \item Automation: Replace manual scheduling with a centralized, conflict-free scheduling system that takes instructor preferences, timetables, and time zones into account. 
    \item AI Integration: Utilize transcripts of past Zoom recordings to train a NLP-based model to automatically evaluate instructor performance based on classroom environment, taking key performance indicators such as volume, pitch, etc into account. 
    \item Integration with external APIs: Enable automated Zoom link generation and integration, along with automated email services to send emails to parents each month regarding class details.  
    \item Analytics Dashboard: Provide visual analytics for instructors and admins to monitor class details,teaching quality, and finance-related matters. 
\end{itemize}

\section{Functional Requirements}

This section details all functions and features of the system. Each requirement is organised into modules with identifiers, priorities, descriptions, detailed requirements, and acceptance criteria.

\subsection{Instructor Management}

\subsubsection{FR-IM-001: Instructor Profile Management}
\textbf{Priority:} High

\textbf{Description:} The system shall maintain comprehensive instructor profiles including availability, expertise, preferences, and performance history.

\textbf{Detailed Requirements:}
\begin{itemize}
    \item Profile includes: name, email, contact information, country, subjects taught, preferred age groups, teaching experience level (XP).
    \item Instructors set weekly availability (day, start time, end time) with support for multiple time slots and unavailable periods.
    \item Instructors define class preferences including minimum/maximum classes per month/day.
    \item Profile automatically tracks teaching history, total classes taught, average ratings, and performance metrics.
    \item Instructors may update availability and preferences, with changes effective in the next scheduling cycle.
    \item Instructors will be able to view their historical performance evaluation data through visual dashboards.
\end{itemize}

\textbf{Acceptance Criteria:}
\begin{itemize}
    \item Profile form validates required fields and data types.
    \item Availability supports multiple time slots per day.
    \item Historical data is immutable and audit-logged.
\end{itemize}

\subsection{Student and Parent Management}

\subsubsection{FR-SPM-001: Student Registration and Profile Management}
\textbf{Priority:} High  

\textbf{Description:} The system shall maintain detailed student profiles linked to parent/guardian accounts.

\textbf{Detailed Requirements:}
\begin{itemize}
    \item Student profile includes: name, date of birth, current course/module.
    \item Parent profile includes: name, email, contact number, country, and subscription plan.
    \item System tracks student progress across modules and courses.
    \item Instructors and admins can view student progress and performance metrics.
\end{itemize}

\textbf{Acceptance Criteria:}
\begin{itemize}
    \item Multiple students can be linked to one parent account.
    \item Automatic and accurate age calculation based on date of birth.
    \item Instructors assigned to a student can access relevant profile data.
\end{itemize}

\subsubsection{FR-SPM-002: Enrollment Management}
\textbf{Priority:} High  

\textbf{Description:} The system shall manage student enrollments including subscription status, attendance, and lesson-level feedback.

\textbf{Detailed Requirements:}
\begin{itemize}
    \item Enrollment links a student to a scheduled class.
    \item Subscription statuses: active, pending, suspended, cancelled.
    \item Attendance recorded for each lesson (present/absent/makeup).
    \item Lesson-level data stored: project links, lesson ratings (0--100\%), assessment scores, instructor comments.
    \item System auto-calculates average lesson rating across lessons.
    \item Next course/module is automatically assigned, however, instructors/ops may override.
\end{itemize}

\textbf{Acceptance Criteria:}
\begin{itemize}
    \item Attendance and performance data editable only by instructors or ops.
    \item Accurate automatic calculation of average lesson rating.
    % \item Notification triggered on subscription status changes.
\end{itemize}

\subsection{Class Scheduling and Assignment}

\subsubsection{FR-CSA-001: Automated Class Scheduling}
\textbf{Priority:} Critical  

\textbf{Description:} The system shall automatically generate monthly class schedules using instructor availability, student demand, and optimization constraints.

\textbf{Detailed Requirements:}
\begin{itemize}
    \item Scheduling algorithm runs monthly after cutoff date OR run manually by Ops.
    \item Algorithm considers:
    \begin{itemize}
        \item Instructor availability and min/max class preferences.
        \item Instructor expertise (subjects, age groups, class size, etc.), XP.
        \item Student preferences (time slots, age, language, class type).
        \item Existing enrollments and progression.
        \item Zoom account availability.
    \end{itemize}
    \item Prevents instructor schedule conflicts.
    \item Optimizes workload balance and student-instructor fit.
    \item Schedule includes class day, time, start date, and frequency.
\end{itemize}

\textbf{Acceptance Criteria:}
\begin{itemize}
    \item No overlapping instructor class times on the same zoom account.
    \item All active students assigned to classes.
    \item Instructor class count within min/max preferences unless admin override.
    \item Ops can manually adjust schedule post-generation.
    % \item Algorithm completes within 2 hours for up to 200 instructors and 2000 students.
\end{itemize}

\section{Non-functional Requirements}

\subsection{Performance Requirements}
\begin{itemize}
    \item \textbf{Response Time:} The system should respond to instructor or admin requests (e.g., loading class data, managing students) within a reasonable time (3-5 seconds) under normal network conditions.
    \item \textbf{Concurrent Users:} The system should support all current instructors concurrently without noticeable performance degradation.
\end{itemize}

\subsection{Reliability \& Availability}
\begin{itemize}
    \item \textbf{Uptime:} The system should maintain availability during the working hours of each instructor.
    \item \textbf{Error Handling:} In case of unexpected errors (e.g., database disconnection), the system should display user-friendly error messages and attempt automatic recovery.
\end{itemize}

\subsection{Security Requirements}
\begin{itemize}
    \item \textbf{Authentication:} Only authorised users (admins and instructors) should be able to log in, verified through stored usernames and passwords.
    \item \textbf{Access Control:} Instructors would primarily have access to their assigned classes (including any classes that they may be substituting), while admins should have full access to all data relevant to their particular role/access level.
    \item \textbf{Data Protection:} Sensitive information (passwords, student data, grades) should be encrypted in storage and during transmission (HTTPS).
\end{itemize}

\subsection{Usability Requirements}
\begin{itemize}
    \item \textbf{Interface Simplicity:} The interface should be intuitive and easy to navigate, requiring minimal training for instructors and admins.
    \item \textbf{Accessibility:} The system should use readable fonts, consistent layouts, and colour contrast suitable for all users.
    \item \textbf{Feedback:} The system should provide clear feedback for user actions (e.g., confirmation messages on successful operations).
\end{itemize}

\subsection{Maintainability}
\begin{itemize}
    \item \textbf{Modular Design:} The codebase should follow a modular architecture (e.g., MVC or component-based structure) to make updates easier.
    \item \textbf{Documentation:} The project should include relevant developer documentation for database schema, APIs, and code structure.
\end{itemize}

\subsection{Portability}
\begin{itemize}
    \item \textbf{Browser Compatibility:} The CMS should function correctly on modern browsers (Chrome, Edge, Firefox, etc.).
    \item \textbf{Device Compatibility:} The system should be optimised for desktop use with a possible mobile view.
\end{itemize}

\subsection{Compliance \& Ethical Considerations}
\begin{itemize}
    \item \textbf{Data Privacy:} The system must comply with general data protection principles (no unauthorised sharing of student information).
    \item \textbf{Content Integrity:} The system must prevent unauthorised alteration or deletion of class records or grades (ideally via efficient design where no user can supersede their roles).
\end{itemize}

\section{External Interfaces}

\subsection{User Interfaces}
The TeachWise platform provides two distinct dashboard interfaces tailored to different user roles: the Instructor Dashboard and the Admin Dashboard.

\subsubsection*{Instructor Dashboard}
The Instructor Dashboard serves as the primary interface for teaching staff to manage their classes, track performance, and configure preferences. The dashboard comprises five main screens:
\begin{enumerate}
    \item \textbf{Class Information Screen:} Displays upcoming and past class schedules, student enrollment details, curriculum materials, and Zoom meeting links. Instructors can view class rosters, zoom links, and access lesson plans.
    \item \textbf{Notifications Panel:} Centralizes all system notifications including class reminders, evaluation report availability, schedule changes, and administrative announcements. Push notifications alert instructors to time-sensitive updates.
    \item \textbf{Settings Screen:} Allows instructors to configure profile information, set weekly availability preferences for the scheduling algorithm, update contact details, and customize notification preferences.
\end{enumerate}

\subsubsection*{Admin Dashboard}
The Admin Dashboard provides comprehensive oversight and control capabilities for platform administrators. It includes four primary functional areas:
\begin{enumerate}
    \item \textbf{Scheduler Screen:} Interfaces with the genetic algorithm-based scheduling system to generate optimized class assignments. Administrators can view proposed schedules, manually adjust assignments when necessary, resolve conflicts, and finalize monthly timetables. The interface displays instructor availability matrices, student preferences, and constraint satisfaction metrics.
    \item \textbf{Evaluations Screen:} Presents aggregated teaching quality assessment data across all instructors. The evaluation results can be sent to instructors via email as well.
    \item \textbf{Finance Screen:} Manages instructor payment processing, tracks class completion rates, calculates monthly compensation based on classes taught, and generates financial reports.
    \item \textbf{Overview Screen:} Provides a comprehensive dashboard displaying key platform metrics including total active instructors, enrolled students, scheduled classes, completion rates, average teaching quality scores, and system health indicators. Interactive visualizations enable drill-down analysis of platform performance.
\end{enumerate}

\subsection{Application Program Interface (API)}
TeachWise integrates with two external APIs to automate critical communication workflows:

\subsubsection*{Zoom API Integration}
The platform leverages the Zoom Video Communications API (v2) to programmatically generate meeting links for scheduled classes. Upon class creation or schedule confirmation, the system makes authenticated requests to the Zoom API endpoint \texttt{/users/\{userId\}/meetings} with parameters specifying:
\begin{itemize}
    \item Meeting topic (class name and session identifier)
    \item Scheduled start time and duration
    \item Meeting settings (waiting room enabled, participant video defaults)
\end{itemize}
The API returns a unique meeting URL and meeting ID, which are automatically displayed on the instructor interface. This eliminates manual meeting creation overhead and ensures consistent meeting configuration across all classes.

\subsubsection*{Gmail API Integration}
The Gmail API (Google Workspace) enables automated email communication for two primary use cases:
\begin{enumerate}
    \item \textbf{Monthly Parent Reports:} At the end of each month, the system generates performance summary emails for parents containing student attendance records, class completion status, and progress indicators. The Gmail API's \texttt{/users/me/messages/send} endpoint is used to dispatch HTML-formatted emails with personalized content.
    \item \textbf{Instructor Evaluation Delivery:} When automated teaching quality assessments are completed, the system compiles evaluation reports and sends them to instructors via email. Reports include PDF attachments with detailed performance analytics and are sent using authenticated SMTP through the Gmail API.
\end{enumerate}
Both API integrations utilize OAuth 2.0 authentication to securely access user accounts and maintain compliance with data privacy requirements.

\subsection{Hardware/Communication Interfaces}
TeachWise operates over standard internet infrastructure and does not require dedicated hardware beyond instructor and admin workstations with modern web browsers. All communication between the frontend, backend, and external services occurs over HTTPS. Real-time data exchange with the Zoom API and Gmail API relies on RESTful HTTP requests. Cloud storage (Dropbox) is accessed via its own API for Zoom recording retrieval and AI evaluation processing.

\section{Datasets}
The TeachWise platform was developed and tested using four primary datasets, each serving distinct components of the system. This section describes the composition, sources, and characteristics of these datasets.

\subsection{Instructor and Student Database}
To simulate the operational environment of an online coding education platform, we constructed a synthetic database modeling real-world instructor and student profiles. The details of these are present in section 4.2. 

\subsection{Audio Recordings for Evaluation System}
The teaching quality evaluation component was trained and tested on 113 audio recordings spanning diverse instructional contexts. The dataset combines proprietary recordings from the partner educational institution with publicly available educational content from YouTube.

\textbf{Partner Institution Recordings (n=45):}
Audio recordings of actual online coding tutoring sessions where instructors teach programming concepts to students in one-on-one or small-group settings. Sessions were recorded with instructor consent and anonymized to protect student privacy. Recordings range from 3--15 minutes in duration and cover topics including:
\begin{itemize}
    \item Python programming fundamentals
    \item Scratch game development
    \item Web development (HTML, CSS, JavaScript)
    \item Data structures and algorithms
\end{itemize}

\textbf{YouTube Educational Videos (n=68):}
Publicly available instructional videos selected based on three criteria: (1) active teaching with live audience interaction, (2) sustained instructional delivery rather than passive lectures, and (3) clear audio quality suitable for prosodic analysis. Videos were segmented into 3--4 minute chunks to create uniform sample lengths. Sources include:
\begin{itemize}
    \item MIT OpenCourseWare lectures (computer science, algorithms)
    \item Abdul Bari algorithm tutorials (binary search, sorting)
    \item ESL/English teaching demonstrations (Teacher Peggy series)
    \item Physics lectures (Walter Lewin)
    \item Economics instruction (circular flow diagrams)
\end{itemize}

\textbf{Audio Specifications:}
\begin{itemize}
    \item Format: WAV (16-bit PCM, 16 kHz sampling rate)
    \item Average duration: 3.4 minutes per segment
\end{itemize}

\subsection{Ground Truth Quality Annotations}
Each audio recording in the evaluation dataset was manually annotated by two experienced educators from the partner institution, both with backgrounds in teacher training and online instruction quality assessment. Annotators independently evaluated each recording on a 5-point Likert scale across four teaching quality dimensions:
\begin{itemize}
    \item \textbf{Energy:} Vocal enthusiasm, dynamism, and instructional intensity (1=very low, 5=very high)
    \item \textbf{Patience:} Calmness, clarity, and supportive tone (1=impatient/rushed, 5=very patient)
    \item \textbf{Student Attention:} Ability to maintain engagement through vocal delivery (1=disengaging, 5=highly engaging)
    \item \textbf{Lesson Pace:} Appropriateness of speaking speed and content progression (1=too slow/fast, 5=optimal)
\end{itemize}
Annotations were collected independently to minimize bias, with no communication between annotators during the evaluation process. 
For curriculum fidelity evaluation, ground truth was derived from predefined lesson plans provided by the partner institution for their proprietary recordings. Each lesson plan contained explicit learning objectives, instructional steps, and expected student outcomes. YouTube recordings did not have corresponding curriculum documentation and were therefore used exclusively for prosodic quality assessment training.

    \item \textbf{Operations Admin:} Adds new students and instructors to the system. Oversees class schedules and instructor-student assignments.

Each backend module communicates through internal APIs and interacts with the database to store or retrieve data.

\begin{itemize}

    \item \textbf{Scheduling Engine:} Prevents conflicts and ensures equitable workload distribution across instructors.
    \item \textbf{Performance Evaluation Service:} Processes Zoom transcripts via the NLP pipeline to assess teaching quality.

    \item \textbf{Notification \& Email Service:} Sends automated notifications (e.g., new class assigned, evaluation completed).

These integrations ensure seamless communication between the teaching environment and the backend automation processes.

    \item \textbf{Payments:} Logs payment calculations, status, and transactions.

\subsection*{Data Flow}
The data flow begins when operations finalize the monthly student list. The Scheduling Engine generates class schedules and the system’s Matching Engine pairs students and instructors, storing assignments in the database.  Corresponding Zoom links are generated and stored via API integration. Instructors access this information through the Instructor Dashboard, while administrators can monitor system-wide metrics in real time via the Admin Dashboard. After each session, Zoom recordings are processed through the AI Evaluation Service, producing performance analytics that are reviewed by the Evaluation Team before being logged into the instructor’s profile. This data then feeds into the Finance Module for payment calculation and bonus allocation.
